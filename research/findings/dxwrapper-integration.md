# dxwrapper integration (D3D8 → D3D9 + windowing)

**Status: SHIPPED (2026-05-04).** Vendored as submodule at `third_party/dxwrapper/`, built from source.

## Why dxwrapper

dgVoodoo 2 is the gold standard for D3D 1-9 → modern translation, but it's **closed source** and was archived January 2026 (last release 2.87.1). Can't vendor or patch.

`elishacloud/dxwrapper` is the open-source equivalent (MIT licensed, actively maintained as of 2026-05). It bundles:
- `crosire/d3d8to9` for D3D8 → D3D9 translation
- A full D3D9 wrapper layer (`m_IDirect3D9Ex`, `m_IDirect3DDevice9Ex`)
- Windowing options (`EnableWindowMode`, `FullscreenWindowMode`, etc.)
- FPS limiter, AA, anisotropic, etc. (we don't use these)

Drop-in: produces `d3d8.dll` (a stub) + `dxwrapper.dll` (the actual library) + `dxwrapper.ini`. Game's import of `d3d8.dll` resolves to our stub, which loads dxwrapper.

## Our config (`Game/dxwrapper.ini`)

Key flags we set (rest defaulted):
```ini
[Compatibility]
D3d8to9                    = 1
EnableD3d9Wrapper          = 1

[d3d9]
EnableWindowMode           = 1
FullscreenWindowMode       = 1
WindowModeBorder           = 0
```

Effect: game's D3D8 `IDirect3D8::CreateDevice` is intercepted by d3d8to9, which calls real `IDirect3D9::CreateDevice` (where **our** mod's hook fires for resolution override). dxwrapper forces windowed mode and removes border, then sizes the window to monitor (after our `SetWindowPos` already did so, the call is idempotent).

## Build instructions (rough)

dxwrapper's `dxwrapper.sln` is a Visual Studio MSBuild solution. Out-of-the-box build had two issues on VS 2026 BuildTools:

### 1. Toolset v143 not installed
The solution targets `v143` (VS 2022). Our system has `v145` (VS 2026 BuildTools). Bulk-replace in all four `.vcxproj` files:
```powershell
Get-ChildItem -Path "third_party\dxwrapper" -Filter "*.vcxproj" -Recurse |
  ForEach-Object { (Get-Content $_.FullName -Raw) -replace 'v143','v145' |
                    Set-Content $_.FullName -NoNewline }
```

### 2. d3d8 PreBuildEvent crashes
The `d3d8/BuildNo.rc` is generated by a chain of `cmd /q /c "echo … >> build.bat"` followed by `cmd /q /c "call build.bat"`. The cmd quoting of `^|`, `^>`, etc. inside echo'd lines breaks under modern cmd, build.bat ends up empty/broken, and `call build.bat` fails.

Fix: replace the entire `d3d8\` block of the PreBuildEvent in `dxwrapper.vcxproj` with a single line that creates a static BuildNo.rc:
```xml
<Command>...keep Dllmain\BuildNo.rc generation...
cmd /q /c "cd /D ""$(ProjectDir)d3d8\"" &amp;&amp; if not exist BuildNo.rc echo #define FILEVERSION 1.15.1.0 &gt;BuildNo.rc"</Command>
```

### 3. Submodule recursion
dxwrapper bundles many sub-submodules (`External/Logging`, `External/Hooking`, `External/d3d8to9`, `External/imgui`, etc.). Must clone with `--recursive`:
```bash
git submodule update --init --recursive
```

### Build command
```powershell
$msbuild = "C:\Program Files (x86)\Microsoft Visual Studio\18\BuildTools\MSBuild\Current\Bin\MSBuild.exe"
& $msbuild "third_party\dxwrapper\dxwrapper.sln" /m /p:Configuration=Release /p:Platform=Win32 /v:minimal
```

Output:
- `third_party/dxwrapper/bin/Release/dxwrapper.dll`
- `third_party/dxwrapper/bin/Release/dx8/d3d8.dll`  (stub renamed)
- `third_party/dxwrapper/bin/Release/dx8/dxwrapper.ini` (template; our deployed version overrides flags)

Deploy to `Game/`:
- `dxwrapper.dll`
- `d3d8.dll` (from `dx8/`)
- `dxwrapper.ini` (our customized version with the four flags above)

## How dxwrapper interacts with our mod

```
Wilbur.exe (D3D8 game)
   └─ imports d3d8.dll
            ↓
   ───── d3d8.dll (dxwrapper stub) loads dxwrapper.dll ─────
            ↓
   dxwrapper hooks d3d8 entry points (Direct3DCreate8 etc.)
            ↓
   d3d8to9 wrapper: IDirect3D8::CreateDevice → IDirect3D9::CreateDevice
                                                       ↓
   Our mtr-asi.asi (loaded by Ultimate ASI Loader as dinput8.dll proxy)
   has hooked IDirect3D9::CreateDevice via vtable[16] of d3d9.dll
            ↓
   Our hook: rewrites pp.BackBufferWidth/Height to monitor + SetWindowPos
            ↓
   Real d3d9.dll IDirect3D9::CreateDevice → real GPU device
            ↓
   dxwrapper wraps the returned device → game gets m_IDirect3DDevice9Ex
            ↓
   Game render calls (BeginScene/Draw/EndScene/etc.) flow through wrapper
   Our hooks on real device's EndScene/Reset/SetTransform fire when wrapper
   forwards
```

Two ASI proxies (`dinput8.dll` for our mod, `d3d8.dll` for dxwrapper) coexist in the same process without conflict because they wrap different DLLs. This is a documented coexistence path in the modding scene; dxwrapper README explicitly mentions Ultimate ASI Loader compatibility.
